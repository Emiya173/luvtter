# 信件应用 功能设计文档

**版本**: v1.0（功能定稿版）
**范围**: 产品功能设计，不含技术实现细节

---

## 一、产品定位

一款以"寄信"为核心交互的慢节奏通信应用。在即时通讯主导的时代，通过真实的等待、不可撤回的郑重、以及手写的温度，让用户重新体验书信带来的仪式感与思考空间。

**核心气质**：
- 功能现代、UI 干净
- 节奏慢、有仪式感
- 拟真的机制，但不做怀旧皮肤
- 无已读、无表情包、无实时对话

**核心用户**：想在快节奏生活中慢下来、重视深度表达的用户。

---

## 二、核心机制

### 2.1 寄信与送达

- 信件从寄出到送达有**真实的等待时间**。
- 信件**一经寄出不可撤回**。
- 送达时间由**距离**和**邮票档位**共同决定。
- 最短送达时间：几小时；最长送达时间：不超过两周。

### 2.2 距离计算

- 内部使用**抽象距离值**（0–1000）作为统一单位。
- **真实地址**按经纬度换算距离值。
- **虚拟地址**由用户以"锚点位置 + 相对距离"的方式设定。
- 距离对寄件人不可见，寄件人只看到预计送达时间。

### 2.3 邮票档位

分四档，决定送达速度：

| 档位 | 描述 |
|------|------|
| 平信 | 最慢 |
| 航空 | 中等 |
| 特快 | 较快 |
| 限时 | 最快（稀有 / 付费） |

邮票除功能外，本身也作为收藏品存在，可有节日限定款、成就解锁款。

### 2.4 不可撤回

信件寄出后无法撤回、无法修改、无法删除（对寄件人而言已离手，对收件人而言即将到来）。这是产品价值点的一部分，强制用户在寄出前慎重考虑。

---

## 三、地址系统

### 3.1 多地址

- 每个用户可设置**多个地址**。
- 所有信件**统一进入同一个收件箱**，地址不分仓。
- 地址独立影响送达时间：寄到哪个地址，距离就按哪个地址计算。

### 3.2 地址类型

- **真实地址**：用户设置时可读取位置，按经纬度换算。
- **虚拟地址**：用户选择一个锚点位置，并设定相对距离（浪漫化地名如"月球背面"等）。

### 3.3 地址规则

- 用户可设置**默认地址**。
- 寄信人默认寄到对方的默认地址，也可选择其他地址。
- 地址**修改只影响未来**的信件；已在路上的信按寄出时地址计算，不受影响。
- 地址对寄件人**不可见**，只显示预计送达时间。

---

## 四、写信体验

### 4.1 输入方式

支持四种输入方式：

1. **键盘输入**
2. **手写输入**（支持触控笔 / 手指书写）
3. **语音输入**（转写为文字）
4. **扫描输入**（拍摄纸质信件）

### 4.2 扫描功能

用户拍摄纸质信件后，可手动选择两种处理方案：

- **保留原貌**：透视矫正 + 背景规整 + 对比度增强 + 轻微风格化。最终为光栅图像，但有"正经扫描件"的质感。
- **重绘到信纸**：提取笔迹轮廓，在应用内标准信纸上重新合成。保留手写形态，去除照片的噪点和光照不均。

两个方案用户手动选择，所见即所得。

### 4.3 信纸样式

- 应用内置多种信纸样式供选择。
- 写信时可为当前这封信选择信纸。
- 扫描"重绘到信纸"时也使用此信纸。

### 4.4 字体

- 字体为**每封信独立选择**（不是账户级偏好）。
- 内置一组精选字体，覆盖不同气质（手写感、衬线、无衬线等）。
- 中文字体按需下载，避免应用体积过大。

### 4.5 涂改机制

这是手写信氛围的关键特性：

- 默认的**删除操作**替换为**划线**（涂改痕迹保留）。
- 划线后的文字**对方可见**。
- 刚输入的文字有**几秒时间窗口**可彻底删除；过窗口后只能划线。
- 支持**主动划线**已写好的文字（"涂改笔"模式）。
- 涂改数据以"文本 + 涂改区间"的自定义结构存储，便于后续搜索与 OCR 索引。

### 4.6 草稿与封存

- 支持保存草稿。
- 支持**封存草稿**：可选的冷静期机制，封存后经过一段时间才能重新打开。作为可选项，用户自行决定是否使用。

---

## 五、附件与信封重量

### 5.1 附件类型（V1）

- **照片**：每封信限制张数与单张尺寸
- **贴纸 / 装饰**：应用内置的图案（花、邮票、蜡封图案等）

录音等其他类型附件留至 V2。

### 5.2 重量机制

- 每个附件占用信封重量。
- 超过当前邮票可承载的重量时，**强制升级邮票档位**。
- 写信界面实时显示当前重量 / 邮票可承载上限。

---

## 六、信件呈现

### 6.1 拆信交互

- 收到信件时有**拆信封动画**。
- 动画为**一次性**：首次打开时播放，之后再打开直接进入信件内容。

### 6.2 信件阅读

- 信件**全屏展示**（手写信需要完整视觉，可缩放）。
- 以**滚动为主**，长信可选**翻页**模式。

---

## 七、拟真事件

低概率触发的非破坏性事件，增加趣味性。**不做丢件等让用户收不到信的机制。**

### 7.1 意外提前

极低概率触发，比预计送达时间更早送达。

### 7.2 信件磨损

- **按在途时间**触发（时间越长概率越高）。
- 分多个等级：轻微折痕、明显褪色、边角破损等。
- 纯视觉效果，不影响内容可读性。

### 7.3 途中明信片

- 长距离 / 长时间在途的信件，途中随机触发小明信片。
- 明信片为系统发送的小通知，告知信件"正在经过某地"或说明延误原因（如"因暴风雪延误 X 天"）等。
- 一封信在途中最多触发有限张，避免打扰。

---

## 八、收件与整理

### 8.1 收件箱

- 所有收到的信件统一进入收件箱（无论寄到哪个地址）。
- 支持按寄件人、时间、主题等维度整理。

### 8.2 寄件箱

- 显示已寄出、在途的信件。
- 在途状态抽象为三档，不显示具体进度：
  - **正在寄出**
  - **路上**
  - **即将到达**

### 8.3 分类与标签

- **分类**：信件分入不同分类
- **标签**：多标签，支持颜色区分
- **收藏夹**：标记重要信件

### 8.4 搜索

- 支持按内容、寄件人、标签、时间等搜索。
- 手写信后台 OCR 生成**搜索索引**，但**展示时仍为原始手写信**（OCR 文字不外显）。

### 8.5 回信关联

- 收到的信与自己的回信关联成"信件往来"线索。
- 可视化采用**信件堆叠**效果（一摞真实信件的感觉），而非 IM 对话气泡。
- 支持同一封信的多次回信。

---

## 九、联系人

### 9.1 地址簿

- 存储联系人：好友、备注、地址、关系标签（家人 / 朋友 / 笔友 / 陌生等）。
- 支持从联系人选择寄信对象。

### 9.2 非好友寄信

- 不加好友也可寄信（通过用户标识）。
- 用户可设置"只接收好友来信"开关（作为隐私保护）。

---

## 十、通知

### 10.1 通知范围

**仅应用内通知**，不做系统级推送。

### 10.2 通知类型

三类通知，用户可**分别开关**：

1. **新信到达**
2. **明信片到达**（途中明信片等拟真事件）
3. **有人回信**

### 10.3 未读提示（待定）

可选：应用图标 Badge 显示未读数。通过应用自身在启动 / 进入后台时更新，不依赖推送通道，不打扰用户但能在自然瞥见手机时发现有新信。

---

## 十一、账号系统

### 11.1 登录方式

- **邮箱**
- **Apple 登录**
- **Google 登录**

### 11.2 新用户引导

首次打开应用，用户地址簿为空，引导流程为：

- **给未来的自己寄第一封信**：写一封延迟送达的信给自己，作为产品体验的起点。既让用户熟悉寄信流程，又是一个有诗意的仪式。

---

## 十二、气质细节

这些细节共同决定产品气质：

- **时间表达**：自然语言（"今天下午""昨天黄昏"），不用"2 小时前"这类
- **数字表达**：偏文学化（"已寄出一百多封"），不精确到个位
- **空状态文案**：如"邮差还在路上"而非"暂无数据"
- **加载动画**：信封飞过的动画，而非 spinner
- **无已读标记**
- **无表情包**，以贴纸替代
- **不显示"对方正在回信"**：保留送达惊喜感

---

## 十三、不做的功能（V1 明确不做）

- **丢件**：不做让用户收不到信的机制
- **撤回 / 修改 / 删除已寄信件**：保持"一经寄出"的严肃性
- **漂流瓶 / 陌生人随机信**：内容审核与安全成本高，延后考虑
- **群发信**：反仪式感
- **实体纸质信寄送**：对接邮政 API 成本高，留至 V2
- **端到端加密**：代价过大，与搜索 / OCR / 找回密码等功能冲突
- **已读标记**：与产品气质冲突
- **"对方正在回信"状态**：破坏等待感
- **系统级推送**：强化"慢"的气质
- **录音附件**（V1 不做，V2 再考虑）

---

## 十四、功能优先级建议

**V1 必做（MVP）**

- 账号系统、地址（单地址起步也可）
- 寄信 / 收信 / 送达时间机制
- 邮票档位（基础四档）
- 键盘输入、手写输入
- 拆信动画、信件展示
- 应用内通知
- 基础收件箱 / 寄件箱

**V1 完整版**

- 多地址、虚拟地址
- 语音输入、扫描输入（两种方案）
- 信纸样式、字体选择
- 涂改机制
- 草稿与封存
- 附件（照片 + 贴纸）、重量机制
- 拟真事件（提前、磨损、途中明信片）
- 分类、标签、收藏、搜索（含 OCR 索引）
- 回信关联、信件堆叠
- 地址簿

**V2 考虑**

- 录音附件
- 实体信寄送
- 漂流瓶 / 笔友发现
- 年度信件回顾
- 邮票商店 / 付费限定款
- 纪念日提醒

## 附录：待后续技术方案阶段确认的点

以下是在产品层面已确定方向、但具体参数留待技术实现时细化的事项：

- 距离值与送达时间的具体映射公式
- 邮票各档位的具体时间系数与随机波动范围
- 信件磨损各等级的触发时间阈值
- 途中明信片的触发距离 / 时间阈值、频次限制
- 附件重量的具体数值（每张照片多少克、每档邮票承重多少）
- 涂改时间窗口的具体秒数
- 信纸、字体的具体款式数量与样式
- 应用图标 Badge 是否启用
