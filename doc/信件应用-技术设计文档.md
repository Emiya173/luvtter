### 1.2 客户端架构

**技术栈**: Compose Multiplatform + Ktor Client + SQLDelight + Koin

**平台覆盖**:

- **Android / iOS / Desktop(JVM)**: 完整功能
- **Web(Wasm)**: 极简方案(只读查看、基础搜索),V1 不含写信/扫描/手写

**KMP 模块结构**:

```
letter-app/
├── api-contract/            # 与服务端共享的 DTO + Api interface
├── shared/
│   ├── domain/              # 纯 Kotlin:实体、值对象、领域规则
│   ├── data/                # Repository、DataSource
│   ├── network/             # Ktor Client 封装
│   ├── database/            # SQLDelight schema + 生成 DAO
│   ├── platform/            # expect/actual 平台能力接口
│   └── ui/                  # Compose 共享 UI
└── composeApp/
    ├── androidApp/
    ├── iosApp/
    ├── desktopApp/
    └── webApp/              # 独立 UI,不复用 shared/ui
```

**模块职责边界**:

| 模块 | 依赖 | 职责 |
|------|------|------|
| domain | stdlib | 纯业务逻辑,零外部依赖,100% 单测 |
| data | domain + network + database | 协调网络与本地存储 |
| network | domain + api-contract | Ktor Client、DTO、SSE 封装 |
| database | 无业务依赖 | SQLDelight,平台 driver 用 expect/actual |
| platform | 无业务依赖 | 相机、文件、语音、生物识别等 expect 接口 |
| ui | domain + data | Compose 共享组件,不直接调平台 API |
| composeApp/* | 全部 shared + platform actual | 应用入口 + 平台实现 |

**必须用 expect/actual 的能力**:

| 能力 | Android | iOS | Desktop | Web |
|------|---------|-----|---------|-----|
| 相机拍摄 | CameraX | AVFoundation | 系统相机/Webcam | MediaDevices API |
| 文件选择 | SAF | UIDocumentPicker | JFileChooser | File API |
| 手写板压感 | MotionEvent | UITouch | AWT MouseEvent | Pointer Events |
| 语音输入 | SpeechRecognizer | SFSpeechRecognizer | 服务端 STT | Web Speech API |
| 生物识别 | BiometricPrompt | LocalAuthentication | OS API | 无 |
| 安全存储 | EncryptedSharedPrefs | Keychain | OS keyring | IndexedDB(弱) |

### 1.3 服务端架构

**单体起步,模块化组织**:

```
server/src/main/kotlin/com/letter/
├── Application.kt           # Ktor 启动入口
├── config/                  # DI、数据库、对象存储配置
├── auth/                    # 登录、JWT、第三方认证
├── user/                    # 用户、地址、联系人
├── mail/                    # 信件核心(寄、收、查询)
├── delivery/                # 送达调度、拟真事件、明信片
├── media/                   # 图像处理调度、对象存储、OCR
├── stamp/                   # 邮票、信纸、贴纸
├── notification/            # 应用内通知 + SSE
├── social/                  # 关系、权限
├── common/                  # 跨模块工具、基础类型
└── db/                      # Exposed 表定义、迁移
```

**分层**: Route → Service → Repository → DB

- Route: 认证、校验、序列化、限流
- Service: 业务编排、事务边界(`newSuspendedTransaction`)
- Repository: 返回 domain 对象而非 `ResultRow`
- DTO(api-contract) 与 Domain 分离

**为什么不上来拆微服务**: MVP 业务边界未稳定,单体 + Postgres 单实例能撑到万级日活,拆分随时可付,合并成本高。

### 1.4 客户端 ↔ 服务端交互

- **主通道**: REST + JSON(Ktor/kotlinx.serialization)
- **通知推送**: SSE(单向,轻量,自动重连,HTTP 原生)
- **离线策略**: 本地 SQLDelight 全量缓存,`updated_at > last_sync` 增量同步,草稿默认本地

### 1.5 架构决策要点

- **单体 + Postgres 单实例起步**: Postgres 功能强大(JSON、全文、地理、任务队列),少组件少故障面
- **Web 延后且极简**: Compose for Web(Wasm)未完全成熟,移动与桌面先打磨
- **图像处理在服务端**: 客户端跨平台实现成本高、设备差异大、迭代慢,服务端统一
- **不在客户端计时送达**: 服务端存 `delivery_at`,客户端查询时过滤,避免时钟不可信

## 二、数据模型

### 2.1 全局约定

- 主键: UUIDv7(自带时间序,利于索引,不暴露规模)
- 时间戳: 所有表有 `created_at` / `updated_at`,类型 `TIMESTAMPTZ`
- 命名: 表名复数小写下划线
- 软删除 vs 物理: 地址/联系人软删,通知/会话物理删,信件永不删
- 字符集: UTF-8

### 2.2 用户与认证

```sql
-- 用户主表
CREATE TABLE users (
  id              UUID PRIMARY KEY,
  handle          VARCHAR(64) UNIQUE NOT NULL,    -- 支持中文,3-20 字符,注册后不可改
  handle_finalized BOOLEAN NOT NULL DEFAULT FALSE, -- 是否已确认 handle
  display_name    VARCHAR(64) NOT NULL,
  avatar_url      TEXT,
  bio             TEXT,
  only_friends    BOOLEAN NOT NULL DEFAULT FALSE,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_users_handle ON users(handle);

-- 认证凭证(一个用户可有多种登录方式)
CREATE TABLE auth_credentials (
  id              UUID PRIMARY KEY,
  user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  provider        VARCHAR(16) NOT NULL,     -- 'email' | 'apple' | 'google'
  identifier      VARCHAR(255) NOT NULL,    -- email / Apple sub / Google sub
  secret_hash     TEXT,                     -- Argon2 哈希(仅 email)
  verified        BOOLEAN NOT NULL DEFAULT FALSE,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(provider, identifier)
);
CREATE INDEX idx_auth_user ON auth_credentials(user_id);

-- 登录会话
CREATE TABLE auth_sessions (
  id              UUID PRIMARY KEY,
  user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  refresh_token   VARCHAR(128) UNIQUE NOT NULL,   -- 建议存哈希
  device_name     VARCHAR(64),
  platform        VARCHAR(16),
  last_active_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
  expires_at      TIMESTAMPTZ NOT NULL,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_sessions_user ON auth_sessions(user_id);
CREATE INDEX idx_sessions_expires ON auth_sessions(expires_at);
```

**Handle 规则**: 支持中文 + 字母 + 数字 + 下划线,3-20 字符,注册后不可改(应用层校验 + 数据库唯一约束)。

### 2.3 地址系统

```sql
CREATE TABLE user_addresses (
  id              UUID PRIMARY KEY,
  user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  label           VARCHAR(32) NOT NULL,
  type            VARCHAR(16) NOT NULL,           -- 'real' | 'virtual'

  -- 真实地址字段
  latitude        DOUBLE PRECISION,
  longitude       DOUBLE PRECISION,
  city            VARCHAR(64),
  country         VARCHAR(64),

  -- 虚拟地址字段
  anchor_id       UUID REFERENCES virtual_anchors(id),  -- 选了预设则非空
  anchor_lat      DOUBLE PRECISION,                      -- 冗余快照,解耦预设表修改
  anchor_lng      DOUBLE PRECISION,
  virtual_distance INTEGER,                              -- 0-1000

  is_default      BOOLEAN NOT NULL DEFAULT FALSE,
  deleted_at      TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_addresses_user ON user_addresses(user_id) WHERE deleted_at IS NULL;
CREATE UNIQUE INDEX idx_addresses_default
  ON user_addresses(user_id)
  WHERE is_default = TRUE AND deleted_at IS NULL;

-- 虚拟地址预设锚点池
CREATE TABLE virtual_anchors (
  id              UUID PRIMARY KEY,
  code            VARCHAR(32) UNIQUE NOT NULL,
  name            VARCHAR(64) NOT NULL,            -- "月球背面""撒哈拉沙漠"
  description     TEXT,
  latitude        DOUBLE PRECISION NOT NULL,
  longitude       DOUBLE PRECISION NOT NULL,
  image_url       TEXT,
  order_index     INTEGER NOT NULL DEFAULT 0,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
```

**设计要点**:

- 真实与虚拟地址合表,用 `type` 区分
- 虚拟地址存冗余 anchor 坐标快照,解耦预设表修改
- 软删除(历史信件仍要显示 label)

### 2.5 邮票、信纸、贴纸

```sql
-- 邮票定义
CREATE TABLE stamps (
  id              UUID PRIMARY KEY,
  code            VARCHAR(32) UNIQUE NOT NULL,
  name            VARCHAR(64) NOT NULL,
  tier            INTEGER NOT NULL,              -- 1=平信 2=航空 3=特快 4=限时
  image_url       TEXT NOT NULL,
  weight_capacity INTEGER NOT NULL,
  speed_factor    DOUBLE PRECISION NOT NULL,     -- 送达时间系数
  is_default      BOOLEAN NOT NULL DEFAULT FALSE,
  is_limited      BOOLEAN NOT NULL DEFAULT FALSE,
  available_from  TIMESTAMPTZ,
  available_to    TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 信纸定义
CREATE TABLE stationeries (
  id              UUID PRIMARY KEY,
  code            VARCHAR(32) UNIQUE NOT NULL,
  name            VARCHAR(64) NOT NULL,
  background_url  TEXT NOT NULL,
  thumbnail_url   TEXT NOT NULL,
  is_default      BOOLEAN NOT NULL DEFAULT FALSE,
  is_limited      BOOLEAN NOT NULL DEFAULT FALSE,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 贴纸定义
CREATE TABLE stickers (
  id              UUID PRIMARY KEY,
  code            VARCHAR(32) UNIQUE NOT NULL,
  name            VARCHAR(64) NOT NULL,
  image_url       TEXT NOT NULL,
  weight          INTEGER NOT NULL,
  is_default      BOOLEAN NOT NULL DEFAULT FALSE,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);

-- 用户拥有的资源(统一表)
CREATE TABLE user_assets (
  id              UUID PRIMARY KEY,
  user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  asset_type      VARCHAR(16) NOT NULL,          -- stamp | stationery | sticker
  asset_id        UUID NOT NULL,
  quantity        INTEGER NOT NULL DEFAULT 1,    -- 邮票消耗,信纸/贴纸恒为 1
  acquired_from   VARCHAR(32),                   -- default | purchase | gift | achievement | daily
  acquired_at     TIMESTAMPTZ NOT NULL DEFAULT now(),
  UNIQUE(user_id, asset_type, asset_id)
);
CREATE INDEX idx_user_assets_user ON user_assets(user_id, asset_type);

-- 每日奖励领取记录
CREATE TABLE daily_rewards (
  user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
  reward_date     DATE NOT NULL,                 -- 按用户时区
  claimed_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  PRIMARY KEY (user_id, reward_date)
);
```

- 邮票消耗型: 寄一封 `quantity -= 1`
- 信纸/贴纸解锁型: `quantity` 恒为 1
- 注册时批量插入默认款(`acquired_from = 'default'`)
- 每日奖励通过 `daily_rewards` 的唯一键保证幂等
- `reward_date` 按**用户时区**计算

### 2.6 信件(核心)

**状态机**:

```
draft → in_transit → delivered → read
         ↑
     [寄出动作: 校验、扣邮票、算距离、生成拟真事件]
```

```sql
CREATE TABLE letters (
  id                  UUID PRIMARY KEY,

  sender_id           UUID REFERENCES users(id),      -- 可空(用户注销)
  recipient_id        UUID REFERENCES users(id),
  recipient_address_id UUID REFERENCES user_addresses(id),
  sender_address_id   UUID REFERENCES user_addresses(id),

  stamp_id            UUID REFERENCES stamps(id),
  stationery_id       UUID REFERENCES stationeries(id),

  status              VARCHAR(16) NOT NULL DEFAULT 'draft',
                      -- draft | in_transit | delivered | read

  sealed_until        TIMESTAMPTZ,                   -- 冷静期截止
  sent_at             TIMESTAMPTZ,                   -- 寄出时间
  delivery_at         TIMESTAMPTZ,                   -- 预计送达
  delivered_at        TIMESTAMPTZ,                   -- 实际送达
  read_at             TIMESTAMPTZ,

  distance_value      INTEGER,                       -- 0-1000 快照
  total_weight        INTEGER NOT NULL DEFAULT 0,

  reply_to_letter_id  UUID REFERENCES letters(id),   -- 回信指向
  wear_level          INTEGER NOT NULL DEFAULT 0,    -- 发送时快照(运行时动态算)

  sender_hidden_at    TIMESTAMPTZ,
  recipient_hidden_at TIMESTAMPTZ,

  created_at          TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at          TIMESTAMPTZ NOT NULL DEFAULT now()
);

CREATE INDEX idx_letters_recipient
  ON letters(recipient_id, delivery_at DESC)
  WHERE status IN ('delivered', 'read') AND recipient_hidden_at IS NULL;

CREATE INDEX idx_letters_sender
  ON letters(sender_id, created_at DESC)
  WHERE sender_hidden_at IS NULL;

CREATE INDEX idx_letters_pending_delivery
  ON letters(delivery_at)
  WHERE status = 'in_transit';

CREATE INDEX idx_letters_reply_to
  ON letters(reply_to_letter_id)
  WHERE reply_to_letter_id IS NOT NULL;
```

```sql
-- 信件内容
CREATE TABLE letter_contents (
  letter_id       UUID PRIMARY KEY REFERENCES letters(id) ON DELETE CASCADE,
  content_type    VARCHAR(16) NOT NULL,             -- text | handwriting | scan
  font_code       VARCHAR(32),

  body_json       JSONB,                            -- text: segments 结构
  body_url        TEXT,                             -- handwriting/scan: S3 URL

  index_text      TEXT,                             -- 搜索用全文(OCR 或 segments 拼接)

```

```json
}
```


### 2.10 异步任务队列(Postgres)

```sql
CREATE TABLE async_tasks (
  id              UUID PRIMARY KEY,
  task_type       VARCHAR(32) NOT NULL,
                  -- process_scan | process_handwriting | ocr_index
                  -- generate_postcard | deliver_letter
  payload         JSONB NOT NULL,
  status          VARCHAR(16) NOT NULL DEFAULT 'pending',
                  -- pending | running | done | failed
  attempts        INTEGER NOT NULL DEFAULT 0,
  max_attempts    INTEGER NOT NULL DEFAULT 3,
  last_error      TEXT,
  scheduled_at    TIMESTAMPTZ NOT NULL DEFAULT now(),
  started_at      TIMESTAMPTZ,
  finished_at     TIMESTAMPTZ,
  created_at      TIMESTAMPTZ NOT NULL DEFAULT now(),
  updated_at      TIMESTAMPTZ NOT NULL DEFAULT now()
);
CREATE INDEX idx_tasks_pending
  ON async_tasks(scheduled_at, id)
  WHERE status = 'pending';
```

Worker 用 `SELECT FOR UPDATE SKIP LOCKED` 抢锁消费。

### 2.11 ER 关系

```
users ──┬── auth_credentials / auth_sessions
        ├── user_addresses ──► virtual_anchors
        ├── contacts / blocks
        ├── user_assets ──► stamps / stationeries / stickers
        ├── daily_rewards
        ├── folders
        ├── favorites
        ├── notifications / user_notification_prefs
        └── letters (sender / recipient)

letters ──┬── letter_contents (1:1)
```

### 2.12 设计取舍摘要

| 决策 | 理由 |
|------|------|
| 信件内容走对象存储 | 大字段不入 DB,备份/迁移/查询更快 |
| `letter_folders` 带 `user_id` | 同一封信对不同用户可归不同分类 |
| 寄出时快照距离 | 地址可变,距离是历史事实 |
| 不用 PostGIS | 距离只算一次,不需空间索引 |
| 永不删信件 | 产品承诺,用软删 `*_hidden_at` 实现对用户隐藏 |

## 四、关键技术点

### 4.1 图像处理管线

**整体流程**:

```
客户端拍照 → 上传原图至 S3(uploads/)
  ↓
服务端异步任务(image-worker,Python)
  ↓
Step 1: 预处理(EXIF 矫正、降采样)
Step 2: 纸张检测 + 透视矫正(OpenCV Canny + findContours + warpPerspective)
Step 3: 光照均衡(shading correction)
  ↓
分支:
  ├── restore(保留原貌): CLAHE 对比度增强 + 轻微纹理
  └── redraw(重绘到信纸): 颜色分离提取笔迹 + 叠加到信纸底图
  ↓
输出 WebP(全尺寸 + 缩略图) → 上传 S3(processed/)
  ↓
并行: OCR 索引 → 更新 letter_contents.index_text / index_tsv
  ↓
更新 async_tasks.status = 'done' → SSE 通知客户端
```

**关键算法选择**:

- **纸张检测**: 长边缩到 1024 做检测,Canny + 最大四边形轮廓,找不到时降级为"用原图"
- **光照均衡**: 灰度大核高斯模糊得背景光照图 → 原图除背景图 → 归一化
- **笔迹提取(MVP)**: HSV 空间,高 V 低 S 判为纸张(alpha=0),其余保留 RGB。边缘 alpha 羽化消除锯齿
- **未来升级**: U-Net 语义分割模型,精度更高但需 GPU + 训练数据

**架构**:

```
ktor-server (Kotlin) ──enqueue──► Postgres queue ◄──poll── image-worker (Python)
                                                                │
                                     ┌──────────────────────────┤
                                     │                          │
                                     ▼                          ▼
                                   MinIO(读原图)              PaddleOCR
                                     │
                                     ▼
                                   MinIO(写结果)
```

Python worker 独立进程/容器,与 Kotlin 通过 Postgres 队列解耦。

### 4.2 手写笔迹数据结构

**存笔画轨迹而非图像**。理由: 矢量无损缩放、支持回放动画、文件小(几 KB-几十 KB)、后处理灵活。

**数据格式**:

```json
{
  "version": 1,
  "canvas": { "width": 1080, "height": 1920, "background": "stationery_linen" },
  "strokes": [
    {
      "id": 0,
      "color": "#222222",
      "width": 3.5,
      "tool": "pen",
      "points": [
        { "x": 120.5, "y": 200.1, "p": 0.6, "t": 0 },
        { "x": 122.3, "y": 201.5, "p": 0.7, "t": 16 }
      ]
    }
  ]
}
```

- `p`: 压感 0-1(支持 Apple Pencil / Surface Pen)
- `t`: 毫秒时间戳(第一个点为 0),用于回放
- `tool`: pen | brush | pencil | highlighter

**优化**: Ramer-Douglas-Peucker 点简化 + 贝塞尔平滑。

**渲染**: Compose Canvas 绘制,压感映射笔粗,多段不同粗细的线段拼接。

**存储**: S3 存 JSON,`letter_contents.body_url` 指向 URL。缩略图客户端生成后上传。

### 4.3 涂改数据结构与渲染

**数据层 segments JSON**(已定义于 2.6)。

**编辑器运行时状态**:

```kotlin
data class EditableSegment(
    val text: String,
    val style: SegmentStyle?,          // null | Strikethrough
    val createdAt: Long,               // 用于时间窗口判断
    val canDelete: Boolean             // 基于 createdAt 计算
)
```

**核心交互逻辑**:

- 输入字符 → 追加到最后 segment(无 style)或新建
- 定时更新 `canDelete`(N 秒后变 false)
- 删除键:
  - `canDelete = true` → 物理删除
  - 否则 → 拆出字符放入 strikethrough segment
- 选中 + 涂改笔按钮 → 选区拆为 strikethrough segment

**渲染**: Compose `AnnotatedString` + `SpanStyle(textDecoration = LineThrough)`。阅读视图直接 `Text(annotated)`;编辑视图需自定义类 TextField 组件。

**MVP 简化**: 时间窗口作为设置中的可选项"书信模式",默认关闭(删除键直接物理删)。避免新手困惑。

### 4.4 距离与送达时间算法

**距离计算**:

地球半周长约 20000km,映射到 0-1000,即 1 距离值 ≈ 20km。

真实 ↔ 真实:
```
D = min(1000, haversine(lat1,lng1,lat2,lng2) / 20)
```

真实 ↔ 虚拟:
```
D = min(1000, haversine(real, anchor) / 20 + virtual_distance)
```

虚拟 ↔ 虚拟:
```
D = min(1000, haversine(anchor1, anchor2) / 20 + v1 + v2)
```

**距离感知对照**:

| D | 大致对应 |
|---|---------|
| 0-50 | 同城 |
| 50-200 | 邻省 |
| 200-500 | 跨国近邻 |
| 500-800 | 跨大洲 |
| 800-1000 | 极远 / 纯虚拟 |

**送达时间分段函数**:

```kotlin
fun baseHours(d: Int): Double = when {
    d < 50   -> 3.0 + d * 0.1                // 3-8h
    d < 200  -> 8.0 + (d - 50) * 0.2         // 8-38h
    d < 500  -> 38.0 + (d - 200) * 0.4       // 38-158h
    d < 800  -> 158.0 + (d - 500) * 0.3      // 158-248h
    else     -> 248.0 + (d - 800) * 0.25     // 248-298h
}
```

**邮票系数**:

```kotlin
val stampFactor = when (tier) {
    1 -> 1.0     // 平信
    2 -> 0.6     // 航空
    3 -> 0.35    // 特快
    4 -> 0.15    // 限时
}
```

**最终公式**:

```kotlin
val randomFactor = 0.9 + Random.nextDouble() * 0.2
val hours = (baseHours(d) * stampFactor * randomFactor)
    .coerceIn(1.0, 14 * 24.0)
val deliveryAt = sentAt + hours.hours
```

**曲线形状**: 近距离快增长,远距离缓增长,符合人的等待感知。具体参数 playtest 调整。

### 4.5 拟真事件触发

寄信时**一次性生成**所有事件,写入 `letter_events` 表,`visible_at` 控制可见时机。

```kotlin
fun generateEvents(letter: Letter): List<LetterEvent> {
    val events = mutableListOf<LetterEvent>()
    val hours = (letter.deliveryAt - letter.sentAt).toHours()

    // 意外提前: 2% 概率, >24h 触发
    if (hours > 24 && Random.nextDouble() < 0.02) {
        val speedup = Random.nextDouble(0.1, 0.3)
        letter.deliveryAt = letter.sentAt + (hours * (1 - speedup)).hours
        events += earlyArrivalEvent(letter)
    }

    // 途中明信片: D > 500 且 > 72h
    if (letter.distanceValue > 500 && hours > 72) {
        val count = when {
            hours > 240 -> 3
            hours > 120 -> 2
            else -> 1
        }
        repeat(count) { i ->
            val ratio = (i + 1) * 1.0 / (count + 1)
            events += postcardEvent(letter, letter.sentAt + (hours * ratio).hours)
        }
    }

    // 磨损: 每 48/120/240h 触发
    listOf(48, 120, 240).forEachIndexed { i, h ->
        if (hours > h) events += wearEvent(letter, i + 1, letter.sentAt + h.hours)
    }

    return events
}
```

**磨损等级动态计算**(运行时):

```kotlin
fun currentWearLevel(letter: Letter): Int {
    val endTime = letter.deliveredAt ?: now()
    val h = (endTime - letter.sentAt).toHours()
    return when {
        h > 240 -> 3
        h > 120 -> 2
        h > 48  -> 1
        else    -> 0
    }
}
```

已送达信件磨损固定(用 `delivered_at`)。

**明信片文案**: 预设模板池 + `{place}` 插入。MVP 用"远方""途中"等模糊表达,未来接入 GeoNames 离线库查真实地名。

### 4.6 OCR 索引与搜索

**OCR 引擎**: PaddleOCR(中文精度好、免费、本地部署)。image-worker 负责调用。

**流程**:

```
信件寄出 → 如果 content_type 为 handwriting/scan → 创建 ocr_index 任务
worker 消费:
  ├── 下载内容(笔画 JSON 渲染成图 / 扫描图像)
  ├── PaddleOCR 提取文字
  └── 更新 letter_contents.index_text / index_tsv
```

文本信寄出时直接拼接 segments 填 `index_text`,不走 OCR。

**搜索 SQL**:

```sql
SELECT l.*, lc.*, ts_headline('simple', lc.index_text, q.query) AS headline
FROM letters l
JOIN letter_contents lc ON l.id = lc.letter_id
CROSS JOIN (SELECT plainto_tsquery('simple', :keyword) AS query) q
WHERE lc.index_tsv @@ q.query
  AND (
    l.sender_id = :user_id
    OR (l.recipient_id = :user_id AND l.delivery_at <= now())
  )
  AND l.recipient_hidden_at IS NULL
  AND l.sender_hidden_at IS NULL
ORDER BY ts_rank(lc.index_tsv, q.query) DESC, l.delivered_at DESC
LIMIT 20;
```

**中文分词(MVP)**: bigram 手工切分,或 `pg_trgm` 扩展(Postgres 自带)。未来上量用 `pg_jieba` 扩展。

**建议 MVP 从 L1 起步**,用户量到万级再 L2。L3 需专职运维。

## 六、开发路线与风险

### 6.1 分阶段路线(2-3 人团队估算)

| 阶段 | 周期 | 内容 |
|------|------|------|
| 1. 基础设施 | 2-3 周 | Gradle 多模块、CI、认证、脚手架、Hello World |
| 2. 核心流程 | 3-4 周 | 用户/地址/联系人 CRUD,单档邮票,文本信寄收读 |
| 3. 写信体验 | 4-6 周 | 手写输入、涂改、字体、信纸、拆信动画 |
| 4. 扫描 + OCR | 3-4 周 | image-worker、透视矫正、笔迹提取、OCR 索引 |
| 5. 完整功能 | 4-6 周 | 多地址、虚拟锚点、邮票多档、每日奖励、附件、拟真事件、回信、通知 |
| 6. 打磨发布 | 4 周 | 性能、bug、上架、内测 |

总计 **20-27 周** 到 V1 发布。

### 6.2 技术风险提示

**Compose Multiplatform 在 iOS 上的成熟度**: 核心稳定,但滚动手感、键盘交互、系统集成仍需调优。团队至少一人熟悉 Swift/iOS 原生。

**图像处理是算力大头**: 4K 照片完整管线 5-15 秒。需要上传限流、队列监控、worker 水平扩展。

**PaddleOCR 模型体积**: 完整模型几百 MB,image-worker 容器较大,配置好 Docker layer 缓存。

**跨平台字体一致性**: 中文字体在各平台渲染有细微差异,必须自带字体文件,反复测试。

**SQLDelight 在 Wasm 上**: Web driver 是 WebWorker 里的 SQL.js,首载 2MB+。Web 极简版可不用 SQLDelight。

**MinIO 单点**: 生产用分布式 MinIO(4+ 节点)或备份到另一对象存储。

### 6.3 全局共识摘要

**产品**: 慢节奏信件应用,现代 UI,拟真机制,四种输入方式,涂改保留,不可撤回,拟真事件不丢件。多地址、邮票分档、附件重量、OCR 搜索、应用内通知、回信堆叠。

**架构**: Kotlin 全栈(Compose Multiplatform + Ktor),单体模块化,Postgres + MinIO + Python image-worker,SSE 通知,SQLDelight 本地。

**数据**: 约 20 张表,覆盖全功能域。

**接口**: REST + SSE,DTO 共享,游标分页,字符串错误码。

**技术细节**: Python + OpenCV 图像处理,笔画 JSON 存手写,segments 结构存涂改,分段函数映射送达时间,寄信时一次生成拟真事件,PaddleOCR 异步建索引。

**部署**: 开发 Docker Compose,生产 L1 单机起步。
