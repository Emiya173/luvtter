## 路线图建议（基于当前进度）

按技术文档 §6.1 的阶段对应：

- **Stage 1（基础设施）**：✅ 完成
- **Stage 2（核心流程：账号 / 地址 / 联系人 / 单档邮票 / 文本信寄收读）**：✅ 服务端完成；客户端 Desktop 已可端到端验证；缺联系人/handle 引导 UI
- **Stage 3（写信体验：手写、涂改、字体、信纸、拆信动画）**：⬜
- **Stage 4（扫描 + OCR）**：⬜（需引入 image-worker）
- **Stage 5（多地址、虚拟锚点、邮票多档、每日奖励、附件、拟真事件、回信、通知）**：✅ 服务端基本就绪（多地址/虚拟锚点/邮票多档/事件/每日奖励/附件/通知/分类/收藏）；缺回信线索专用 API、搜索 API、通知 SSE、客户端附件/分类 UI
- **Stage 6（打磨发布）**：⬜

### 短期（建议下一两轮迭代）

1. 客户端：联系人界面 + handle 引导 + 草稿管理 + 附件/贴纸 UI + 分类/收藏 UI + 涂改的多 segment 编辑
2. 服务端：search API（tsvector + bigram）、回信 thread API、letter events 独立 API、notifications SSE 长连接
3. 工程：Koin 模块化重构 + testcontainers 集成测试 1~2 个关键路径（auth、send→inbox）

### 中期

4. Stage 3 写信体验：手写笔画数据结构 + 字体下载 + 信纸渲染
5. image-worker（Python + OpenCV + PaddleOCR）容器与 `/uploads/scan` 接口

### 长期

6. Android / iOS 适配与 SQLDelight 本地化
7. Stage 6 打磨：自然语言时间、信封 loading、拆信动画、Badge
